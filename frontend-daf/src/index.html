Below we outline the major modifications of the PR: * **Message flow**: x/btcstaking `MsgCreateFinalityProvider` now handles all finality provider registrations instead of forwarding consumer BSN ones to the x/btcstkconsumer module. The corresponding fuzz test has been updated to reflect the full test case. * **Storage**: the following storage modifications wewre made: * Removed the `fpPk => bsnId` mapping from the x/btcstkconsumer storage. * Consumer finality providers are now stored as their Babylon Genesis peers on x/btcstaking. * To facilitate "Give me all fps securing a BSN ID" queries (which are common ones) we introduce a secondary index for fast lookup by BSN ID: `bsn_id || fp_btc_pk -> nil (just existence marker)`. * This allows for limited overhead for getting all `fp_btc_pk` for a `bsn_id` using an index store. * **Queries**: Moved all x/btcstkconsumer finality provider queries to x/btcstaking. * This only required the modification of the `QueryFinalityProviders` query to now support querying by BSN ID.     Replace toEngine channel with a dedicated subscription API

    yacovm merged to ava-labs/avalanchego at 2025-07-07 20:35:27

    This commit removes the toEngine channel pattern from all VMs and adds an additional API: ``` SubscribeToEvents(ctx context.Context) Message) ``` Which returns when an event (pending transactions or state sync done Message) is sent by the VM. Unlike the toEngine pattern which only notifies the consumer a single time until a block is built, successive calls to `SubscribeToEvents` will return that transactions are pending as long as the mem-pool is not empty and a block can be built, otherwise the calls block. The API change is needed to support consensus protocols which facilitate censorship resistance as part of their protocol and therefore need information about whether a block should be built or not on demand, in contrast to the curren toEngine pattern which notifies the consensus when new transactions arrive only once, and then block building may even fail. Having such an API on the VM side (and not implementing the logic in consensus) forces the VM to handle any side effects of block building (such as re-introducing transactions into the mempool).
